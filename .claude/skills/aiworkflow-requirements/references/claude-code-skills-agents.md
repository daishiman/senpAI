## 目的

- Taskを「一時的ワーカー」として起動するときの **仕様書**
- Task内の試行錯誤ログをメインへ持ち込まない前提で、**入出力と制約**に集中させる

---

## agents/ の位置づけ（誤解防止）

- `skills/.../agents/*.md` は **「サブエージェント（永続人格）の定義」ではない**
- ここに書くのは **Taskの仕様書**（役割・入力・出力・制約・参照先）
- **実行直前に、そのTask分だけまとめて読まれる前提**
  → だから _軽量に_、知識は `references/` に逃がす

---

## agents/*.md 標準フォーマット（必須テンプレ）

`skills/<skill-name>/agents/*.md` は **Task仕様書**であり、各Taskを「一時的ワーカー」として起動する際の **役割・判断基準・入出力契約**を定義する。

**実在する専門家名の扱い**:

- 「思考様式・判断軸の参照ラベル」として扱う
- 本人を名乗る／本人の発言を捏造する／個人情報を扱う、はしない
- 参照するのは **フレームワーク・方法論・思考の型**のみ

**長さ制約**:

- agents/*.md は **実行直前に一括ロード**される前提のため、過度に長文化しない
- 長い知識本文・サンプル・詳細手順は `references/` に外部化し、ここは「参照先」と「適用方法（短文）」に留める

## agents/*.md テンプレ（Markdown見出しで構造化）

以下のテンプレを **各Taskファイル**で採用する（例：`agents/research.md`、`agents/draft.md` など）。

### テンプレート構造概要

Task仕様書は、以下の6つの主要セクションで構成される。各セクションは見出しレベルで明確に区分され、Task起動時に一括ロードされることを前提とした軽量な記述を行う。

| セクション       | 見出しレベル | 必須 | 概要                                   |
| ---------------- | ------------ | ---- | -------------------------------------- |
| Task仕様書タイトル | H1           | Yes  | Task名を明記（例: Task仕様書：Research） |
| メタ情報         | H2           | Yes  | 参照する専門家名（思考様式ラベル）     |
| プロフィール     | H2           | Yes  | 背景・目的・責務の定義                 |
| 知識ベース       | H2           | Yes  | 参考文献と適用方法                     |
| 実行仕様         | H2           | Yes  | 思考プロセス・チェックリスト・制約     |
| インターフェース | H2           | Yes  | 入力・出力の契約定義                   |

---

### セクション詳細：メタ情報

Task仕様書の冒頭に配置し、このTaskが参照する思考様式を明示する。

| 項目   | 記述内容                               | 備考                                   |
| ------ | -------------------------------------- | -------------------------------------- |
| 名前   | 実在する専門家の名前                   | 思考様式の参照ラベルとして使用         |
| 注記   | 本人を名乗らず、方法論のみ適用する旨   | 必須の注意書き                         |

---

### セクション詳細：プロフィール

Taskの役割と期待される成果を定義する。3つのサブセクションで構成される。

| サブセクション | 記述内容                                         | 記述量目安 |
| -------------- | ------------------------------------------------ | ---------- |
| 背景           | なぜこの人物が適しているか。経歴・専門性を記述   | 2-3文      |
| 目的           | 達成すべき具体的なゴール                         | 1-2文      |
| 責務           | 責任範囲と成果物                                 | 1-3文      |

---

### セクション詳細：知識ベース

参考文献セクションとして、このTaskが活用する書籍・フレームワークを列挙する。

**参考文献の記述形式**（書籍ごとに以下の項目を記載）:

| 項目     | 記述内容                                                       | 制約                                   |
| -------- | -------------------------------------------------------------- | -------------------------------------- |
| 書籍     | 書籍名                                                         | 正式名称を使用                         |
| 適用方法 | この書籍の考え方・フレームワークをどのように適用するか         | 短文で記述。詳細はreferences/に外部化  |

**ルール**: 適用方法は短く記述し、詳細な知識本文・サンプル・手順は `references/` ディレクトリに外部化して相対パスで参照すること。

---

### セクション詳細：実行仕様

Taskの実行方法を3つの観点から定義する。

#### 思考プロセス

ステップ番号と具体的な実行内容を順序立てて記述する。

| 要素         | 記述形式                   | 例                                     |
| ------------ | -------------------------- | -------------------------------------- |
| ステップ番号 | 1, 2, 3, ... N             | 順序を明確にする連番                   |
| ステップ内容 | 具体的なアクションを記述   | 「入力データの妥当性を検証する」       |

#### チェックリスト

品質確認のための検証項目と合格基準を定義する。

| 項目         | 記述内容                                           | 必須項目例                                   |
| ------------ | -------------------------------------------------- | -------------------------------------------- |
| 確認項目     | 検証対象の名称                                     | 出力検証、事実確認                           |
| 基準         | 具体的な合格条件                                   | 必須フィールドをリスト化                     |

**推奨チェック項目**:
- 出力検証: すべての必須項目が含まれているか（基準: 必須フィールドをリスト化）
- 事実確認: 推測を事実として述べていないか（基準: 不確実な情報には限定詞を使用。例: 「可能性がある」「推測」「現時点では」）

#### ビジネスルール（制約）

Taskの実行に関する制約条件を列挙する。実行順序、タイミング、スコープなどの制約を明記する。

---

### セクション詳細：インターフェース

Taskの入出力契約を定義する。データの流れと検証ルールを明確化する。

#### 入力定義

各入力データについて以下の項目を定義する。

| 項目         | 記述内容                                           | 例                                           |
| ------------ | -------------------------------------------------- | -------------------------------------------- |
| データ名     | 入力データの識別名                                 | ユーザー要求、コンテキスト情報               |
| 提供元       | データの出所（専門家名または「外部」）             | ResearchAgent、外部                          |
| 検証ルール   | どのような入力を受け入れるか                       | JSON形式、必須フィールドの存在確認           |
| 拒否すべき入力 | どのような入力を拒否するか                       | 空文字、不正なフォーマット                   |
| 欠損時処理   | 入力が欠損した場合の振る舞い                       | デフォルト値使用、前Taskに再要求、エスカレーション |

#### 出力定義

各成果物について以下の項目を定義する。

| 項目           | 記述内容                                         | 例                                           |
| -------------- | ------------------------------------------------ | -------------------------------------------- |
| 成果物名       | 出力データの識別名                               | 調査レポート、ドラフト文書                   |
| 受領先         | データの受け取り先（専門家名）                   | DraftAgent、ReviewAgent                      |
| 出力テンプレート | 変数を用いたデータ出力形式                     | title: {{title}}, content: {{content}}       |
| 内容           | 何を含むか簡潔に説明                             | 調査結果のサマリーと詳細データ               |
