## scripts/ ディレクトリ仕様

### 使用場面

| 場面                         | 説明                   |
| ---------------------------- | ---------------------- |
| 同じコードを繰り返し書き直す | スクリプト化して再利用 |
| 決定論的な信頼性が必要       | AIの不確実性を排除     |
| 複雑な処理の確実な実行       | 冪等性を保証           |

### スクリプト実装要件

| 要件       | 説明                   |
| ---------- | ---------------------- |
| 冪等性     | 何度実行しても同じ結果 |
| エラー出力 | stderr に詳細を出す    |
| 引数検証   | 必須引数を確認         |
| ヘルプ     | `-h`/`--help`          |
| 終了コード | 成功0、一般1、引数2…   |

### 終了コード規則

| 終了コード | 意味           |
| ---------- | -------------- |
| 0          | 成功           |
| 1          | 一般的エラー   |
| 2          | 引数エラー     |
| 3          | ファイル不在   |
| 4          | 検証失敗       |
| 5          | 依存関係エラー |

### 標準スクリプト: log_usage.js

フィードバックループの核。詳細は 7章参照。

## references/ ディレクトリ仕様

### 役割（定義の明確化）

- references/ は **知識の外部ストレージ**
- SKILL.md や agents/\*.md を軽量に保つための逃がし先
- 「毎回読む前提」ではなく **必要時にだけ読む** ことを前提に設計

### 使用場面

| 場面                 | パス形式                                 |
| -------------------- | ---------------------------------------- |
| データベーススキーマ | `references/{{schema-name}}.md`          |
| APIドキュメント      | `references/{{api-name}}.md`             |
| ドメイン知識         | `references/{{domain-name}}.md`          |
| ポリシー・ルール     | `references/{{policy-name}}.md`          |
| ワークフローガイド   | `references/{{workflow-name}}.md`        |
| 出力パターン         | `references/{{output-type}}-patterns.md` |

### ベストプラクティス

| プラクティス             | 説明                                      |
| ------------------------ | ----------------------------------------- |
| SKILL.md を軽量に保つ    | 詳細は references/                        |
| 重複を避ける             | 情報は SKILL.md か references/ のどちらか |
| 大きなファイル（>10k語） | SKILL.md に検索パターンを含める           |
| 100行超のファイル        | 先頭に目次                                |

## Progressive Disclosure パターン

### パターン1: 高レベルガイドと参照

SKILL.mdは「Quick start」で基本的な使用方法を提供し、「Advanced features」で詳細を references/ にリンクする構成とする。

| セクション        | 内容                                                   |
| ----------------- | ------------------------------------------------------ |
| Quick start       | 基本的な使用方法を簡潔に記述                           |
| Advanced features | 各機能名と references/ へのリンクをリスト形式で提供    |

機能リンクは「機能名: See references/機能名.md」の形式で、必要な機能のみ参照読み込みを促す。

### パターン2: ドメイン別構成

ドメイン（機能領域）ごとに参照ファイルを分割する構成。SKILL.mdは概要とナビゲーションに特化する。

| パス                          | 役割                 |
| ----------------------------- | -------------------- |
| skill-name/SKILL.md           | 概要とナビゲーション |
| skill-name/references/        | ドメイン別詳細格納   |
| references/domain-1.md        | ドメイン1の詳細説明  |
| references/domain-2.md        | ドメイン2の詳細説明  |
| references/domain-n.md        | ドメインNの詳細説明  |

### パターン3: バリアント別構成

バリアント（処理パターンの違い）ごとに参照ファイルを分割する構成。SKILL.mdでワークフローとバリアント選択ロジックを提供する。

| パス                          | 役割                     |
| ----------------------------- | ------------------------ |
| skill-name/SKILL.md           | ワークフロー・選択ロジック |
| skill-name/references/        | バリアント別パターン格納 |
| references/variant-1.md       | バリアント1のパターン    |
| references/variant-2.md       | バリアント2のパターン    |
| references/variant-n.md       | バリアントNのパターン    |

**重要なガイドライン**:

- 深くネストした参照を避ける（SKILL.md から1レベルに保つ）
- すべての参照ファイルは SKILL.md から直接リンク（相対パス使用）

### パターン4: family file + archive 構成

large reference skill では、1枚の巨大ドキュメントを以下へ分割する。

| パス | 役割 |
| --- | --- |
| `references/topic-index.md` | family index |
| `references/topic-detail-a.md` | detail |
| `references/topic-detail-b.md` | detail |
| `references/logs-archive-index.md` | archive index |
| `references/logs-archive-YYYY-MM.md` | 月次 archive |

`SKILL.md` は上記すべてへ直リンクを持ち、`LOGS.md` は rolling log に留める。

## assets/ ディレクトリ仕様

コンテキストに読み込むためではなく、Claude が生成する出力で使用するファイル。

### 使用場面

| 場面             | パス形式                           |
| ---------------- | ---------------------------------- |
| テンプレート     | `assets/{{template-name}}.{{ext}}` |
| 画像・アイコン   | `assets/{{image-name}}.{{ext}}`    |
| ボイラープレート | `assets/{{boilerplate-name}}/`     |
| フォント         | `assets/{{font-name}}.{{ext}}`     |
| サンプルデータ   | `assets/{{data-name}}.{{ext}}`     |

### 利点

- 出力リソースをドキュメントから分離
- コンテキストに読み込まずにファイルを使用可能

## ワークフローパターン

### シーケンシャルワークフロー

複雑なタスクでは、操作を明確な順序のステップに分解する。SKILL.md の冒頭でプロセスの概要を示す。

**記述形式**: 「タスク名は以下のステップで行う」で始め、番号付きリストでステップを記述する。

| ステップ構成要素 | 説明                           | 例                              |
| ---------------- | ------------------------------ | ------------------------------- |
| ステップ番号     | 1から順に番号付け              | 1, 2, 3...                      |
| ステップ説明     | 何を行うかの説明               | ファイルを生成する              |
| 実行方法         | スクリプト実行または編集       | run scripts/xxx.js, edit xxx.json |

各ステップは「説明（実行方法）」の形式で記述し、スクリプト実行は「run scripts/スクリプト名」、ファイル編集は「edit ファイル名」と表記する。

### 条件分岐ワークフロー

分岐ロジックを持つタスクでは、Claudeを判断ポイントに導く構成とする。

**記述形式**: 最初に判断ポイントを提示し、条件ごとに分岐先を明示する。

| 構成要素       | 記述方法                                       |
| -------------- | ---------------------------------------------- |
| 判断ステップ   | 「判断ポイントを判断する」と記述               |
| 条件分岐       | 「条件？ → ワークフロー名に従う」形式で列挙    |
| ワークフロー詳細 | 各ワークフロー名と references/ へのリンクを記述 |

条件分岐は太字で条件を示し、矢印（→）で遷移先のワークフロー名を指定する。ワークフローの詳細は references/ 内の個別ファイルに記載し、SKILL.mdからリンクする。

## 出力パターン

### テンプレートパターン

出力形式のテンプレートを提供。厳格さのレベルを要件に合わせる。

**厳格な要件の場合（APIレスポンスやデータ形式）**:

出力構造を厳密に定義し、「必ず以下の正確なテンプレート構造を使用する」と明記する。

| セクション     | 内容形式                     | 用途                 |
| -------------- | ---------------------------- | -------------------- |
| 出力タイプ構造 | 見出しで構造を宣言           | テンプレートの名称   |
| タイトル       | h1見出しで変数を配置         | 出力物のタイトル     |
| セクション1    | 文章形式で内容説明           | 概要・説明セクション |
| セクション2    | 箇条書きで項目と補足情報     | リスト形式の情報     |
| セクション3    | 番号付きリストでアクション項目 | 手順・アクション     |

各セクション名は変数（二重波括弧）で表記し、実際の出力時に適切な名称に置換される。

**柔軟なガイダンスの場合（適応が有用な場合）**:

「以下は妥当なデフォルト形式。最善の判断で使用する」と前置きし、Claudeに適応の余地を与える。

| セクション     | 内容形式               | 適応性                   |
| -------------- | ---------------------- | ------------------------ |
| 出力タイプ構造 | 見出しで構造を宣言     | デフォルト形式として提示 |
| タイトル       | h1見出しで変数を配置   | 状況に応じて調整可       |
| セクション1-2  | 基本構造のみ提示       | コンテキストに応じて適応 |
| セクション3    | 調整の指針を記述       | 調整可能な部分を明示     |
| 適応ガイドライン | 全体的な適応方針を記述 | 判断基準を提供           |

厳格な要件と異なり、セクション内容は概要のみ示し、具体的な記述はコンテキストに委ねる。

### 例示パターン

出力品質が例を見ることに依存する場合、入力/出力のペアを提供する。

**記述形式**: 「以下のパターンに従って出力物を生成する」で始め、パターンと形式ルールを定義する。

| 構成要素   | 内容                             |
| ---------- | -------------------------------- |
| 出力形式名 | h2見出しで形式の名称を宣言       |
| パターン   | 入力と出力の対応関係を示す       |
| 入力       | 入力の説明（変数形式で記述）     |
| 出力       | 出力フォーマット（変数形式で記述）|
| 形式ルール | 箇条書きでルールを列挙           |

**注意**: 具体的な例示はAIの出力を例に引っ張る傾向があるため、パターンと形式ルールで表現する。具体例ではなく、抽象的なパターンを示すことで、Claudeが状況に応じた適切な出力を生成できる。
