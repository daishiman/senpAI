### CI/CD

GitHub Actionsを使用してCI/CDパイプラインを構築する。

#### パイプライン構成（実装済み 2026-01-05）

CI/CDパイプラインは以下の順序で実行される。push/PRをトリガーとして、まず4つのジョブが並列実行され、その後coverageとbuildが依存関係に従って実行される。

**パイプラインフロー**:

| ステップ | ジョブ    | 並列実行 | 依存関係              | 説明                            |
| -------- | --------- | -------- | --------------------- | ------------------------------- |
| 1        | push/PR   | -        | -                     | トリガーイベント                |
| 2        | lint      | 並列     | push/PR               | コード品質チェック              |
| 2        | typecheck | 並列     | push/PR               | TypeScript型チェック            |
| 2        | test      | 並列     | push/PR               | テスト実行・カバレッジ計測      |
| 2        | security  | 並列     | push/PR               | セキュリティスキャン            |
| 3        | coverage  | 単独     | test                  | Codecovへのレポートアップロード |
| 4        | build     | 単独     | lint, typecheck, test | 本番用ビルド実行                |

**Lintジョブ**:
チェックアウト後、pnpmでインストールし、lintとtypecheckを実行する。

**Testジョブ**:
チェックアウト後、pnpmでインストールし、カバレッジ付きテストを実行。

**Coverageジョブ（実装済み 2026-01-05）**:
testジョブ完了後、カバレッジレポートをCodecovにアップロードし、品質ゲートとして機能する。

| 設定項目         | 値             | 設計理由                                   |
| ---------------- | -------------- | ------------------------------------------ |
| 依存関係         | needs: [test]  | testジョブのカバレッジデータを再利用       |
| 実行条件         | PR or main     | featureブランチはPRで検証するため不要      |
| タイムアウト     | 10分           | 通常5分以内で完了、余裕を持って設定        |
| fail_ci_if_error | true           | 品質ゲートとして機能させるため             |
| Codecov Action   | v5             | 最新安定版、セキュリティ修正を自動受信     |
| フラグ           | shared,desktop | パッケージ別レポート、問題箇所の特定が容易 |

**Codecov統合設計（codecov.yml）**:

| 設定項目                | 値   | 採用理由                                       |
| ----------------------- | ---- | ---------------------------------------------- |
| project.target          | 80%  | 業界標準、現実的な達成目標、既存設定との整合性 |
| patch.target            | 80%  | PRでの品質維持、新規コードも同基準適用         |
| threshold               | 1%   | カバレッジ計算の誤差を考慮（79.5%は許容）      |
| comment.require_changes | true | 変更がある場合のみコメント、ノイズ削減         |

**なぜfail_ci_if_error: trueか**:

- カバレッジ低下を確実にブロック（品質ゲート）
- PRマージ前に問題を検出
- 技術的負債の蓄積を防止
- Codecovサービス障害時はCI失敗（一時的にfalseに変更可能）

### ハンドラ単位カバレッジ判定ルール（Phase 7拡張）

IPCハンドラファイルのカバレッジ判定は、ファイル全体ではなくハンドラ単位で評価する。

| ルール | 条件 | 判定 |
|---|---|---|
| Rule-1 | 対象ハンドラの Line >= 80%, Branch >= 60%, Function >= 80% | PASS |
| Rule-2 | ファイル全体が低値でも修正対象ハンドラが基準超過 | PASS |
| Rule-3 | 未カバーのハンドラは未タスク候補として記録 | N/A |
| Rule-4 | ファイル全体の Branch >= 60% | 参考値 |

**P41注記**: v8カバレッジプロバイダはインラインarrow function（例: `getAllowedWindows: () => [mainWindow]`）を独立関数としてカウントする。Function Coverage が実態より低く表示される場合がある。

**測定ツール**: `npx tsx scripts/coverage-by-handler.ts --file <path>`

### 測定コマンド

| 目的                     | コマンド                                              |
| ------------------------ | ----------------------------------------------------- |
| ユニットテストカバレッジ | pnpm vitest --coverage                                |
| 閾値チェック（CI用）     | pnpm vitest --coverage --coverage.thresholds.lines=70 |
| E2Eカバレッジ            | pnpm exec playwright test --reporter=html             |
