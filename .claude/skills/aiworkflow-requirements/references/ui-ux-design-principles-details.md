## UXデザイン法則

### Fitts's Law（フィッツの法則）

**原則**: ターゲットに到達する時間は、ターゲットまでの距離とターゲットのサイズに依存する。

| 適用 | 実装ガイドライン |
|------|-----------------|
| ボタンサイズ | タッチターゲットは最小44px × 44px（Apple HIG準拠） |
| 重要アクション | 画面の端や角に配置（無限の深さを活用） |
| 近接配置 | 関連するアクションは近くに配置 |
| 破壊的アクション | 頻繁に使うボタンから離して配置 |

### Hick's Law（ヒックの法則）

**原則**: 選択肢の数が増えると、意思決定にかかる時間が対数的に増加する。

| 適用 | 実装ガイドライン |
|------|-----------------|
| メニュー項目 | 一度に表示する選択肢は7±2個以内 |
| オンボーディング | 段階的に機能を紹介（Progressive Disclosure） |
| 設定画面 | カテゴリ分けして選択肢をグループ化 |
| デフォルト値 | 推奨オプションを事前選択 |

### Jakob's Law（ヤコブの法則）

**原則**: ユーザーは他のサイトで多くの時間を費やすため、同じように動作することを期待する。

| 適用 | 実装ガイドライン |
|------|-----------------|
| ナビゲーション | 標準的なパターン（サイドバー、タブ）を使用 |
| アイコン | 一般的に認知されたアイコンを使用（🔍検索、⚙️設定） |
| 操作パターン | macOS/Windows標準のキーボードショートカット |
| フォーム | 標準的なフィールド順序（名前→メール→パスワード） |

### Miller's Law（ミラーの法則）

**原則**: 短期記憶で保持できる情報は7±2チャンク。

| 適用 | 実装ガイドライン |
|------|-----------------|
| ナビゲーション | トップレベル項目は5-9個以内 |
| ステップ表示 | ウィザードは5-7ステップ以内 |
| リスト表示 | チャンキング（グループ化）で認知負荷軽減 |
| 情報の提示 | 重要な情報を先に、詳細は展開可能に |

### Gestalt Principles（ゲシュタルト原則）

| 原則 | 説明 | UI適用例 |
|------|------|----------|
| **近接** | 近い要素は関連があると認識 | フォームフィールドのグループ化 |
| **類似** | 似た要素は同じグループと認識 | 同じスタイルのボタン群 |
| **閉合** | 不完全な形を補完して認識 | ローディングリングの進捗 |
| **連続** | 連続した線や形を追う傾向 | ステッパー、パンくずリスト |
| **共通運命** | 同じ動きをする要素は関連と認識 | 一括選択時のハイライト |
| **図と地** | 前景と背景を区別 | モーダルのオーバーレイ |

### Progressive Disclosure（段階的開示）

**原則**: 必要な情報を必要な時に表示し、複雑さを管理する。

| レベル | 表示内容 | 実装例 |
|--------|----------|--------|
| Level 1 | 最も使用頻度の高い機能 | メインツールバー |
| Level 2 | 展開で表示される追加機能 | 「詳細設定」セクション |
| Level 3 | 特定条件で必要な機能 | コンテキストメニュー |
| Level 4 | 上級者向け機能 | 開発者オプション |

### Doherty Threshold（ドハティ閾値）

**原則**: システム応答時間が400ms以下であれば、ユーザーの生産性と満足度が大幅に向上する。

| 応答時間 | ユーザー体験 | 対策 |
|----------|--------------|------|
| 0-100ms | 即時と感じる | 単純な操作はこの範囲で |
| 100-300ms | 少し遅れを感じる | インタラクティブ操作の目標 |
| 300-1000ms | 明確に遅れを認識 | ローディング表示を検討 |
| 1000ms+ | フロー状態が途切れる | プログレス表示必須 |

### Goal-Gradient Effect（目標勾配効果）

**原則**: 目標に近づくほど、達成への動機が高まる。

| 適用 | 実装ガイドライン |
|------|-----------------|
| プログレスバー | 開始時点で少し進んだ状態から表示 |
| オンボーディング | 完了ステップ数を可視化 |
| フォーム入力 | 入力済み項目数を表示 |
| ゲーミフィケーション | 達成度バッジ、レベル表示 |

### Aesthetic-Usability Effect（美的ユーザビリティ効果）

**原則**: 美しいデザインは、より使いやすく、エラーに寛容に感じられる。

| 適用 | 実装ガイドライン |
|------|-----------------|
| ビジュアルデザイン | 一貫したカラーパレット、タイポグラフィ |
| マイクロインタラクション | 細部のアニメーションへの配慮 |
| ホワイトスペース | 適切な余白で視覚的な呼吸を確保 |
| 一貫性 | UIパターンの統一で信頼感を醸成 |

### Tesler's Law（テスラーの法則）

**原則**: 複雑さには限界があり、完全に排除することはできない。誰かが複雑さを引き受ける必要がある。

| 判断基準 | アプローチ |
|----------|-----------|
| ユーザーの専門性が低い場合 | システム側で複雑さを吸収 |
| 頻繁に使う機能 | シンプルなデフォルトを提供 |
| パワーユーザー向け | カスタマイズオプションを提供 |
| 本質的な複雑さ | ユーザー教育、ヘルプを充実 |

### Peak-End Rule（ピーク・エンドの法則）

**原則**: 体験の評価は、ピーク時（最も強烈な瞬間）と終了時の印象で決まる。

| フェーズ | 設計ガイドライン |
|----------|-----------------|
| ピーク体験 | 感動的な成功アニメーション、達成感の演出 |
| エラー時 | 温かみのあるエラーメッセージ、回復支援 |
| 終了時 | 保存完了の明確な確認、次回への期待感 |
| 離脱防止 | 未保存警告を親切なトーンで |

## 認知負荷の軽減

### 視覚的階層

| 要素 | 優先度の表現方法 |
|------|-----------------|
| サイズ | 重要度に応じてフォントサイズを変える |
| 色 | プライマリカラーで重要なアクションを強調 |
| 位置 | 重要な情報は左上から右下への視線の流れに沿って |
| 余白 | 重要な要素の周りに十分な余白を確保 |

### 認知負荷の種類と対策

| 種類 | 説明 | 対策 |
|------|------|------|
| **内在的** | タスク自体の複雑さ | 段階的開示、チャンキング |
| **外在的** | UIの複雑さ | シンプルなデザイン、一貫性 |
| **本質的** | 学習に必要な負荷 | ツールチップ、ヘルプ、チュートリアル |

## Tap & Discover 哲学

> 参照: [00-ui-design-foundation.md](../../../docs/30-workflows/skill-import-agent-system/tasks/ui-overhaul/00-ui-design-foundation.md) Task 5C, 5D

### 設計思想

```
任天堂のゲームUI: 最初の操作は1つだけ。やるうちに自然にわかる
iPhoneのように:   何も読まなくても触っているうちにわかる
LINEのように:     会話が中心。フォームやテーブルは最小限
```

### 3原則

| 原則 | 説明 | 実装ガイドライン |
|------|------|-----------------|
| **Level 1 = 4個以下** | 最初に見えるのは最大4つの大きな要素 | ヒーローコンテンツ、プライマリCTA、ナビゲーション |
| **Level 2 = タップで詳細** | 複雑さは全てタップ/クリック後に表示 | アコーディオン、ボトムシート、スライドインパネル |
| **全操作にフィードバック** | タップ→バウンス、成功→チェック、失敗→シェイク | CSS変数による統一アニメーション |

### マイクロインタラクション仕様

#### 追加CSS変数

| 変数名 | 値 | 用途 |
|--------|-----|------|
| `--ease-bounce` | `cubic-bezier(0.34, 1.56, 0.64, 1)` | バウンス感のある跳ね返り |
| `--ease-anticipate` | `cubic-bezier(0.68, -0.55, 0.27, 1.55)` | 溜めてから跳ねる |
| `--scale-hover` | `1.02` | ホバー時の微拡大 |
| `--scale-active` | `0.97` | タップ/クリック時の微縮小 |
| `--scale-bounce` | `1.05` | 成功時のバウンスピーク |

#### タッチターゲット最小サイズ

| 要素タイプ | 最小サイズ | 推奨サイズ | 根拠 |
|------------|-----------|-----------|------|
| ボタン | 44 × 44px | 48 × 48px | Apple HIG タッチターゲット |
| カード | 80 × 80px | 120 × 120px | 指での選択しやすさ |
| リスト項目 | 56px（高さ） | 64px（高さ） | 行間タップの誤操作防止 |
| アイコンボタン | 44 × 44px | 44 × 44px | Apple HIG 最小タップ領域 |
| チップ/バッジ | 36 × 36px | 40 × 40px | 小要素でもタップ可能 |
| サジェスチョンバブル | 48px（高さ） | 56px（高さ） | テキスト付きのタップ領域 |

#### 標準フィードバックパターン

| 操作 | CSS | 用途 |
|------|-----|------|
| ホバー | `transform: scale(var(--scale-hover)); transition: var(--duration-fast) var(--ease-out)` | カード、ボタン |
| アクティブ/タップ | `transform: scale(var(--scale-active)); transition: 50ms ease-in` | 全インタラクティブ要素 |
| 成功 | `scale(1) → scale(var(--scale-bounce)) → scale(1)` 300ms `var(--ease-bounce)` | 追加完了、送信完了 |
| 失敗/エラー | `translateX(-4px, 4px, -4px, 4px, 0)` 400ms (シェイク) | バリデーションエラー |
| 出現 | `opacity: 0 → 1` + `translateY(8px → 0)` 200ms `var(--ease-out)` | カード出現、リスト項目追加 |
| 消失 | `opacity: 1 → 0` + `scale(1 → 0.95)` 150ms `ease-in` | 削除、ポップオーバー閉じ |

### UX言語ガイドライン

素人ユーザーが直感的に理解できるよう、画面上の全テキストは技術用語を平易な日本語に変換する。

#### 技術用語→やさしい日本語 変換テーブル

| 技術用語 | やさしい日本語 | 使用例 |
|----------|---------------|--------|
| エージェント | AIアシスタント | 「AIアシスタントが作業を代行します」 |
| スキル | できること / ツール | 「新しいツールを追加する」 |
| パーミッション | 許可 / できること | 「AIにできることを設定する」 |
| バリデーション | 確認 / チェック | 「入力内容を確認しています」 |
| IPC | （表示しない） | ユーザーに見せない |
| インポート | 追加する | 「ツールを追加する」 |
| エクスポート | 書き出す | 「データを書き出す」 |
| プロバイダ | AI | 「AIを選ぶ」 |
| モデル | AIの種類 | 「使うAIの種類を選ぶ」 |
| ストリーミング | （表示しない） | 処理中のアニメーションで代替 |
| リフレッシュ | 更新する | 「最新の情報に更新する」 |
| ワークスペース | 作業スペース | 「作業スペースを開く」 |
| ダッシュボード | ホーム | 「ホームに戻る」 |
| セッション | やりとり | 「前回のやりとりを続ける」 |
| コンテキスト | 状況 / 背景情報 | 「ファイルの背景情報をAIに伝える」 |
| フィルター | しぼり込み | 「条件でしぼり込む」 |
| ページネーション | （表示しない） | 「もっと見る」ボタンで代替 |
| トースト/通知 | お知らせ | 「お知らせが届きました」 |

#### 画面タイトル変換

| 既存タイトル | 新タイトル |
|-------------|-----------|
| ダッシュボード | ホーム |
| エージェントビュー | AIアシスタント |
| スキルセンター | ツールを探す |
| 横断履歴検索ビュー | あなたの記録 |
| 通知センター | お知らせ |
| オンボーディング | はじめよう |
| ワークスペース | 作業スペース |

#### UXライティング原則

1. **主語はユーザー**: 「ファイルがインポートされました」→「ファイルを追加しました」
2. **能動態を使う**: 「設定が保存されました」→「設定を保存しました」
3. **1文15文字以内**: ボタンラベル、見出しは短く
4. **「〜してみよう」で誘う**: 命令形（「〜してください」）ではなく提案形
5. **エラー時は解決策を示す**: 「エラーが発生しました」→「うまくいきませんでした。もう一度試してみてください」

### SuggestionBubble パターン

タップで試す用のバブルUI。チャット/ダッシュボード/オンボーディング画面で共通利用する。

| プロパティ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `label` | `string` | ✅ | バブルに表示するテキスト |
| `icon` | `string` | - | Lucide アイコン名 or 絵文字 |
| `onClick` | `() => void` | ✅ | タップ時のコールバック |
| `size` | `"sm" \| "md" \| "lg"` | - | サイズバリアント（デフォルト: `"md"`） |

**サイズ別仕様:**

| サイズ | 高さ | padding | font-size | 用途 |
|--------|------|---------|-----------|------|
| `sm` | 36px | 8px 16px | 13px | インラインサジェスト |
| `md` | 48px | 12px 20px | 14px | ゼロステート、チャット |
| `lg` | 56px | 16px 24px | 15px | オンボーディング |

### EmptyState mood バリアント

EmptyStateコンポーネントに `mood` プロパティを追加し、画面の文脈に応じた感情トーンを表現する。

| mood | 用途 | アイコンスタイル | テキストトーン |
|------|------|----------------|---------------|
| `welcoming` | 初回訪問、ゼロステート | 暖色系、手招きアイコン | 「ようこそ！」「はじめてみよう」 |
| `encouraging` | 空のリスト、検索結果なし | 中性色、前向きアイコン | 「まだ記録はありません」「試してみよう」 |
| `celebrating` | 全完了、全既読 | 成功色、祝福アイコン | 「おめでとう！」「完了しました 🎉」 |
