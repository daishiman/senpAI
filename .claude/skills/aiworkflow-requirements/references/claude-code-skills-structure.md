## Skill構造仕様

### 大規模 skill docs の分割パターン

500 行超の reference や log を持つ skill では、以下を標準とする。

| 要素 | 標準 |
| --- | --- |
| `SKILL.md` | 入口、quick start、resource navigation、validator command のみを保持 |
| `references/` | family index + detail file へ分割し、1 hop で到達できる導線を持つ |
| `LOGS.md` | rolling log のみを保持し、月次 archive へ退避する |
| root policy | `.claude/skills/...` を canonical、`.agents/skills/...` を mirror とする |
| validation | `quick_validate.js`、`validate_all.js`、`diff -qr` を最低セットとする |

### 標準フォルダ構造（改訂：agents/ を追加）

スキルディレクトリ（skill-name/）は以下の構成要素で構成される。

| パス                      | 必須/推奨/任意 | 説明                                             |
| ------------------------- | -------------- | ------------------------------------------------ |
| SKILL.md                  | 必須           | スキルの中枢ドキュメント（500行以内推奨）        |
| agents/                   | 推奨           | Task仕様書を格納（実行直前にロードされる前提）   |
| agents/task-1.md          | -              | 個別のタスク仕様書                               |
| agents/task-n.md          | -              | 追加のタスク仕様書（必要に応じて複数作成可）     |
| scripts/                  | 任意           | 実行可能コードを格納（Python/Bash/Node等）       |
| scripts/log_usage.js      | 推奨           | フィードバックループ用標準スクリプト             |
| references/               | 任意           | 必要時に読む参照資料（知識の外部化）             |
| assets/                   | 任意           | 出力で使用する素材（テンプレート/雛形等）        |

**重要**: 以下のファイルは作成しない（SkillはAI実行のための情報のみ）

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- その他の補助ドキュメント

## SKILL.md 仕様

### YAML Frontmatter

**公式フィールド（Anthropic準拠）**:

| フィールド      | 型       | 必須 | 説明                                       |
| --------------- | -------- | ---- | ------------------------------------------ |
| `name`          | string   | 必須 | スキル識別子（ハイフンケース、最大64文字） |
| `description`   | string   | 必須 | スキル説明と発動条件（最大1024文字）       |
| `license`       | string   | 任意 | ライセンス情報                             |
| `allowed-tools` | string[] | 任意 | 許可ツール                                 |
| `metadata`      | object   | 任意 | 追加メタ                                   |

**プロジェクト拡張フィールド（改訂）**:

| フィールド     | 型       | 必須 | 説明                                                |
| -------------- | -------- | ---- | --------------------------------------------------- |
| `tags`         | string[] | 任意 | 検索用タグ                                          |
| `dependencies` | string[] | 任意 | 依存スキルのパス（※参照のみ。自動実行を意味しない） |

> **削除**: `references:` フィールド（書籍等の構造化リスト）は本プロジェクトでは採用しない。
> 理由：description 内にアンカーとして統合し、トリガー性能（発見性）を優先するため。

### name フィールドの規則

| 規則                     | 説明                         |
| ------------------------ | ---------------------------- |
| ハイフンケース           | 小文字、数字、ハイフンのみ   |
| 最大64文字               | 長すぎる名前は不可           |
| 先頭・末尾にハイフン不可 | `my-skill` ○、`-my-skill-` × |
| 連続ハイフン不可         | `my-skill` ○、`my--skill` ×  |
| ディレクトリ名と一致     | フォルダ名と完全一致         |

### description フィールド（最重要：トリガー設計）

**descriptionはスキルの主要なトリガーメカニズム。**
Skillは段階的ロードされるため、発見フェーズでは本文より description が効く。

| 規則                          | 説明                                                                 |
| ----------------------------- | -------------------------------------------------------------------- |
| 最大1024文字                  | 簡潔かつ包括的に                                                     |
| 角括弧禁止                    | `<` `>` は使用不可                                                   |
| 「何を」と「いつ」を含める    | 機能＋発動条件を必ず書く                                             |
| 本文に When to Use 不要       | 発動条件は description に集約                                        |
| アンカーは description に統合 | ドキュメント/リファレンス/スイーズ情報は **Markdown禁止で** 埋め込む |

**description 記述形式（Markdown禁止版）**

descriptionフィールドは複数行文字列（YAML パイプ記法）で記述する。箇条書きには `-` や `*` を使わず、行区切りと中黒（•）で表現する。

**構成要素**:

| 要素     | 必須 | 記述内容                                                   |
| -------- | ---- | ---------------------------------------------------------- |
| 概要     | 必須 | スキルの説明（2〜3行）                                     |
| Anchors  | 推奨 | 参照資料のリスト（アンカー名 / 適用: 適用範囲 / 目的: 目的の形式で記述） |
| Trigger  | 必須 | 英語での発動条件（短く、具体例語彙を含める）               |

**Anchorsセクションの記述形式**:

各アンカーは「• アンカー名 / 適用: 適用範囲 / 目的: 目的」の形式で1行ずつ記述する。アンカーには「書籍」「社内ルール」「スキーマ」「Runbook」「API仕様」「設計方針」などを混在させてよい。ただし「適用範囲」と「目的」は必ず添えること。

**Triggerセクションの記述形式**:

「Use when」で始まり、英語で発動条件を記述する。具体的な語彙（キーワード）を含めることで、発見性を高める。

### 本文（Markdown Body）

Frontmatter の後に続く指示とガイダンス。スキル発動後にのみ読まれる。

**推奨セクション構造**:

| セクション              | 必須 | 説明                                         |
| ----------------------- | ---- | -------------------------------------------- |
| 概要                    | 必須 | 1-2文で目的                                  |
| ワークフロー            | 推奨 | Phase ベースの流れ（Task分割の案内を含める） |
| Task仕様（ナビ）        | 必須 | agents/ の各Taskをいつ呼ぶか、入出力は何か   |
| ベストプラクティス      | 必須 | すべきこと/避けるべきこと                    |
| リソース/スクリプト参照 | 推奨 | references/ scripts/ assets/ への導線        |

**行数制約**: 500行以内。超過時は references/ に分離。
