## L-SC07-001: generationMode のローカル state 管理と UI 切替の分離

| 項目         | 内容                                                                                                                                                                                                                  |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 課題         | 当初 `generationMode` は `DescribeStep` の Props として外部から注入する設計だったが、ウィザード全体のフロー分岐（Step 0 の表示切替・Step 2 の onCancel / onRetry 切替など）に影響するため、どこで保持すべきか混乱した |
| 解決策       | `generationMode` を `SkillCreateWizard` の `useState` として保持し、Step 0 のラジオボタン UI で切替。全ステップのハンドラ分岐（handleLlmGenerate / handleGenerate）もここで一本管理する                               |
| 将来への知見 | ウィザード全体に影響する分岐フラグはオーケストレーターコンポーネント（SkillCreateWizard）に置く。子コンポーネントには「今何をすべきか」の Props のみ渡し、モード判定ロジックを持たせない                              |
| 関連タスク   | TASK-SC-07                                                                                                                                                                                                            |

## L-SC07-002: executePlan の skillSpec 必須化（C-1 回避）

| 項目         | 内容                                                                                                                                                                                                             |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 課題         | `executePlan(planId, skillSpec)` の `skillSpec` を optional にすると、Main 側がスキル生成の内容を受け取れないまま空実行してしまうリスクがあった。レビュー段階で「C-1: skillSpec は必須」として明示が必要とされた |
| 解決策       | Renderer 側で `skillSpec` が空の場合は `executePlan` を呼ばず、`setGenerationErrorMsg("実行するスキル仕様がありません")` を返す。型定義でも `skillSpec: string`（非 optional）として固定                         |
| 将来への知見 | IPC 呼び出しの必須引数は型定義と呼び出し元の両方でガードする。型を optional にした時点で「空のまま呼ばれる」コードパスが合法になるため、ドメインとして必須な引数は型レベルで non-optional にする                 |
| 関連タスク   | TASK-SC-07, TASK-SC-06-UI-RUNTIME-CONNECTION                                                                                                                                                                     |

## L-SC07-003: Hybrid State Pattern における対称クリアの必要性

| 項目       | 内容                                                                                                                                                                                                                                                  |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 課題       | `localPlanResult`（ローカル useState）と `currentPlanResult`（Store）の両方に PlanResult を保持する Hybrid State Pattern では、キャンセル時に一方だけクリアすると古い値が残り、次回実行時に意図しない UI が表示される不具合が発生した                 |
| 解決策     | キャンセル系ハンドラ（`handleCancelPlan` / `resetGeneratedState`）では必ず `setLocalPlanResult(null)` と `clearGenerationState()` を対称に呼ぶ。`invalidateGenerationRequests()` も同時に呼び、飛んできた古いレスポンスを request-id ガードで破棄する |
| 標準ルール | Hybrid State Pattern（ローカル + Store 二重保持）では「クリアは常に対称」を設計原則とする。どちらか片方のみのクリアは許可しない                                                                                                                       |
| 関連タスク | TASK-SC-07, TASK-SDK-04-U2                                                                                                                                                                                                                            |

## L-SC07-004: request-id ガードによる遅延レスポンスの破棄

| 項目         | 内容                                                                                                                                                                                                                                                                  |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 課題         | LLM 生成中にユーザーがキャンセルして Step 0 に戻り、再度生成を開始した場合、旧リクエストのレスポンスが後から到着して新しい生成の状態を上書きする問題が起きた                                                                                                          |
| 解決策       | `llmGenerationRequestIdRef` / `templateGenerationRequestIdRef` の2本の ref を用意し、キャンセル時に `invalidateGenerationRequests()` でどちらも同時インクリメント。各 async 関数の冒頭で `const requestId = ...Ref.current` を取得し、完了時に比較して古ければ return |
| 将来への知見 | LLM / template の2経路が同一コンポーネントに共存する場合、request-id ref は経路ごとに分ける。単一 ref での管理は「もう片方のキャンセルで ref が進んでしまう」問題を引き起こす                                                                                         |
| 関連タスク   | TASK-SC-07                                                                                                                                                                                                                                                            |

## L-SC07-005: getWorkflowState による snapshot 再読込パターン

| 項目         | 内容                                                                                                                                                                                                                                                      |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 課題         | `executePlan()` の戻り値だけでは、terminal_handoff が発生したか・verify が失敗したか・persistResult（skillPath）がどこかを判断しきれなかった。レスポンス構造が複数の union 型（成功 / terminal_handoff / エラー応答）を持つため、解釈ロジックが肥大化した |
| 解決策       | `executePlan()` 成功後に `getWorkflowState(planId)` を呼び出し、engine が保持する snapshot を正本として取得する。snapshot の `handoffBundle` / `verifyResult.status` / `persistResult.skillPath` をそれぞれ確認し、UI の分岐に使う                        |
| 将来への知見 | IPC レスポンスの union 型が複雑になったら、snapshot 再読込パターンで engine 側の正本に委譲する。レスポンス型の肥大化は「Renderer がビジネスロジックを持ちすぎ」のサイン                                                                                   |
| 関連タスク   | TASK-SC-07, TASK-SDK-04                                                                                                                                                                                                                                   |

## L-SC07-006: smartDefaults の推論と Q5 外部ツール解決の分離

| 項目         | 内容                                                                                                                                                                                                                                                       |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 課題         | Step 0 で推論した `smartDefaults.tool`（Slack / GitHub / Notion）と、Step 1 の Q5 回答（selectedOption / freeText）から確定する `externalToolName` は別概念だが、当初は一本化しようとして混乱した                                                          |
| 解決策       | `inferSmartDefaults(formData)` は Step 0 入力から「Q1〜Q6 の初期値候補」を返す純粋関数として分離。`resolveExternalIntegration(q5Answer, smartDefaultTool)` は Step 1 完了後に Q5 の確定回答を正本として扱い、`smartDefaultTool` はフォールバックとして使う |
| 将来への知見 | 「推論（初期値）」と「ユーザー確定（正本）」は別の状態として扱う。推論結果を正本扱いするとユーザーが変更しても上書きされる問題が起きる                                                                                                                     |
| 関連タスク   | TASK-SC-07, UT-SKILL-WIZARD-W2-seq-03a                                                                                                                                                                                                                     |

## L-SC07-007: DescribeStep の deprecated 管理と SkillInfoStep への移行

| 項目         | 内容                                                                                                                                                                                                                                                             |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 課題         | 旧 `DescribeStep.tsx` は TASK-10A-C 時点での Step 0 コンポーネントだったが、W1-par-02a で `SkillInfoStep` に置き換えられた後も、`DescribeStep.tsx` 自体は削除されず残留した。仕様書には「DescribeStep」と書かれているが実体は `SkillInfoStep` という乖離が生じた |
| 解決策       | `DescribeStep.tsx` に `@deprecated` JSDoc コメントを追加し、仕様書の「Step 0 コンポーネント」表記を `SkillInfoStep` に統一。削除は「全呼び出し元が切替済み」を確認してから独立タスクとして実施する（委譲パターンと同様の段階的廃止）                             |
| 将来への知見 | コンポーネントのリネーム・置き換えは「旧ファイルを即削除」ではなく「deprecated コメント → 仕様書更新 → 削除タスク登録」の3ステップで行う。即削除は参照元の洗い出し漏れを引き起こす                                                                               |
| 関連タスク   | TASK-SC-07, UT-SKILL-WIZARD-W1-par-02a, UT-SKILL-WIZARD-W2-seq-03b                                                                                                                                                                                               |

## L-SC07-008: generationLockRef による二重実行防止

| 項目       | 内容                                                                                                                                                                                                                                     |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 課題       | LLM 生成ボタンや「実行する」ボタンを高速で複数回クリックした場合、`isGenerating` state の更新が React のバッチ処理で遅延し、2回目の呼び出しが通り抜けてしまうリスクがあった                                                              |
| 解決策     | `generationLockRef = useRef(false)` を全ハンドラで共有する単一の排他ロックとして使用。`handleLlmGenerate` / `handleExecutePlan` / `handleGenerate` それぞれの先頭で `if (generationLockRef.current) return` を実行し、同期的にガードする |
| 標準ルール | L-RT04-001 と同パターン: 非同期処理の排他制御は `useRef` の同期フラグで行う。`useState` だけのガードは React のバッチ更新で無効化される場合がある                                                                                        |
| 関連タスク | TASK-SC-07, TASK-RT-04-AUTHKEY-COMPONENT-DEDUP-001                                                                                                                                                                                       |
