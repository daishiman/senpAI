## テスト数記載基準

Phase 7/9/10/12 でテスト数を記載する際は、以下の基準を遵守すること:

| ルール | 説明 |
| ------ | ---- |
| **実測値のみ使用** | `pnpm test` の実行結果から取得した実際の数値のみ記載する。推定値・概算値は使用しない |
| **実行コマンドを明記** | テスト数を取得したコマンド（例: `pnpm vitest run --reporter=verbose`）を成果物に記録する |
| **タイムスタンプを付与** | テスト実行日時を記録し、数値の鮮度を保証する |

> **注意**: テスト数は実装の進行に伴い変動する。Phase間で数値が異なる場合は、各Phase実行時点の実測値を記載し、差分の理由を明記すること。

## 大規模ファイルのカバレッジ評価ガイドライン（Phase 7）

ファイル全体が500行を超える場合、ファイル全体のカバレッジ数値が不当に低く見える可能性がある。この場合は「新規追加コード部分」を個別に評価する。

| 評価対象 | 判定基準 |
| -------- | -------- |
| ファイル全体のカバレッジ | 参考値として記録する（合否判定には使わない） |
| 新規追加コード部分のBranch Coverage | 70%以上を目標とする |
| 新規追加コード部分のLine Coverage | 80%以上を必須とする |

**評価手順**:
1. `git diff HEAD~1 -- <ファイルパス>` で新規追加行を特定する
2. 追加した関数・分岐ブロックをリストアップする
3. テストコードが追加行の各分岐を網羅しているか確認する

> 注意: ファイル全体のカバレッジ基準（Line 80%, Branch 60%）は小〜中規模ファイルの基準。大規模ファイルでの新規コード追加時は本ガイドラインを優先する。

## プロジェクト全体グローバル閾値が失敗する場合の対処（Phase 7）

`pnpm test:coverage` はプロジェクト全体のカバレッジを集計するため、既存ファイルの未カバー部分が
原因でグローバル閾値を下回り Phase 7 が失敗することがある。
この場合は **対象ファイルを絞り込んで個別計測する**。

### 個別ファイルカバレッジ計測コマンド

```bash
# 単一ファイルのカバレッジを計測（プロジェクト全体閾値の影響を受けない）
pnpm vitest run --coverage --coverage.include='**/targetFile.ts' --coverage.include='**/__tests__/targetFile.test.ts'

# 複数ファイル（ディレクトリ単位）
pnpm vitest run --coverage --coverage.include='**/services/runtime/sdkMessage*.ts'
```

### 判定フロー

| 状況 | 対処 |
| ---- | ---- |
| `pnpm test:coverage` がグローバル閾値超過で失敗 | 対象ファイルを `--coverage.include` で絞り込んで個別計測 |
| 個別計測が閾値を満たす | Phase 7 PASS として扱い、Phase 8 へ進む（グローバル閾値失敗は既存負債として別タスクに委ねる） |
| 個別計測も閾値を満たさない | Phase 6 へ戻りテスト追加 |

> 注意: グローバル閾値の達成は本タスクの新規コードのみでは担保できない（既存負債）。
> Phase 7 は「今回追加したコードのカバレッジ」を確認する目的で実施する。
