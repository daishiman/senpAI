# UIスクリーンショット検証手順

## 概要

UIタスクでスクリーンショットが添付される場合の、取得日・ファイル実在・内容一致を検証する手順。

## 適用判定

タスク分類が以下のいずれかに該当する場合、本手順を**必須**で実施する:

- 分類に `UI` を含む（例: TASK-UI-xx）
- 実装対象に Renderer コンポーネント（`.tsx`）を含む
- Phase 11 手動テストにUI操作を含む

上記に該当しない場合でも、current workflow が `spec_created` / docs-heavy で upstream UI surface の統合再確認を求められているなら、本手順を適用する。完全に非UIで画面確認要求もない場合のみ `N/A（UIタスクではないため）` として記録し、本手順をスキップする。

## ファイル命名規則

```
{task-id}_{screen-name}_{YYYY-MM-DD}.png
```

例: `UT-IMP-PHASE12-TWO_main-view_2026-03-03.png`

## 検証手順

### S-1: ファイル実在確認

```bash
ls -la <screenshotPath>
```

`fs.existsSync(filePath)` による検証も可。

**判定基準**: ファイルが存在すれば OK、存在しなければ NG。

### S-2: 取得日確認

```bash
# macOS
stat -f "%Sm" <screenshotPath>

# Linux
stat -c "%y" <screenshotPath>
```

`fs.statSync(filePath).mtime` でファイル更新日時を取得。

**記録形式**: ISO 8601形式（例: `2026-03-03T14:30:00+09:00`）

### S-3: 取得日の合理性確認

取得日が以下の条件を満たすことを確認する:

- Phase 11 実行日以降であること
- タスクブランチ作成日以降であること
- 未来の日付でないこと

**判定基準**: 上記3条件を全て満たせば OK。

### S-4: 内容目視確認

スクリーンショットの内容が該当画面と一致するかを目視で確認する。

確認項目:

- 画面タイトルが期待と一致する
- 操作対象のUI要素が表示されている
- エラー表示がないこと（正常系の場合）
- テスト対象の状態変化が反映されている（状態変化テストの場合）

**判定基準**: 画面内容が期待と一致すれば OK。

## 証跡テンプレートへの記録

パス・取得日・検証結果を証跡テンプレートのスクリーンショット証跡セクションに転記する。

```markdown
| #   | チェック項目     | コマンド/確認方法              | 結果    |
| --- | ---------------- | ------------------------------ | ------- |
| S-1 | ファイル実在     | ls -la <path>                  | OK      |
| S-2 | 取得日確認       | stat -f "%Sm" <path>           | 2026-03-03 |
| S-3 | 取得日が合理的か | ブランチ作成日以降             | OK      |
| S-4 | 内容目視確認     | 該当画面と一致                 | OK      |
```

## N/A の場合の記録

UIタスクではない場合、以下のように記録する。ただし、ユーザーが branch 全体の画面検証を明示要求した場合はこの N/A 例を使わず、representative screenshot を current workflow 配下へ追加する:

```markdown
### 判定: N/A（UIタスクではないため）

本タスクはUIコンポーネントの実装を含まないため、スクリーンショット検証をスキップする。
```
