## メタ情報フォーマット規約（重複禁止）

未タスク指示書の `## メタ情報` は **1セクションのみ** とする。
`issue_number` の YAML ブロックとメタ情報テーブルを併記する場合も、同じ `## メタ情報` セクション内にまとめる。

| NG例 | OK例 |
| --- | --- |
| `## メタ情報` を2回定義（YAML用 + テーブル用） | `## メタ情報` を1回定義し、その直下に YAML とテーブルを連続配置 |

チェック方法（推奨）:

```bash
rg -n "^## メタ情報" docs/30-workflows/unassigned-task/*.md
# 各ファイルで 1件のみであることを確認
```

## 記録対象の判定

### 記録すべき課題

| 発見元       | 記録対象                                 |
| ------------ | ---------------------------------------- |
| Phase 3      | MINOR判定された設計上の未対応項目        |
| Phase 10     | MINOR判定された実装上の未対応項目        |
| Phase 11     | 手動テストで発見されたスコープ外の改善点 |
| Phase 12     | ドキュメント更新時に発見された追加タスク |
| 全Phase共通  | 将来的に必要となる拡張機能               |
| 全Phase共通  | 技術的負債として認識された項目           |
| 全Phase共通  | セキュリティ上の懸念（緊急性が低いもの） |
| 全Phase共通  | パフォーマンス改善の余地                 |
| コードベース | TODO/FIXME/HACK/XXXコメント              |

### 記録すべきでない課題

| 種別                   | 対応方法                   |
| ---------------------- | -------------------------- |
| 緊急の不具合           | 現タスク内で即時対応       |
| 重大なセキュリティ問題 | エスカレーション＆即時対応 |
| 現スコープ内の修正     | Phase内で対応              |
| MAJOR判定の設計問題    | 該当Phaseへ戻り対応        |

## 品質基準: Why/What/How

### Why（なぜ必要か）

| チェック項目       | 説明                                         |
| ------------------ | -------------------------------------------- |
| 背景が明確         | このタスクが必要になった経緯が説明されている |
| 問題点が具体的     | 現状の問題が定量的/定性的に説明されている    |
| 放置時の影響が記載 | 対応しない場合のリスクが明記されている       |

### What（何を達成するか）

| チェック項目         | 説明                               |
| -------------------- | ---------------------------------- |
| 目的が具体的         | 達成すべきことが一意に解釈できる   |
| 最終ゴールが検証可能 | 完了を判定できる基準がある         |
| スコープが明確       | 含む/含まないが明記されている      |
| 成果物が列挙         | 生成される成果物がリストされている |

### How（どのように実行するか）

| チェック項目       | 説明                               |
| ------------------ | ---------------------------------- |
| 前提条件が明記     | タスク開始前に満たすべき条件がある |
| 実行手順が具体的   | 手順が明確に記載されている         |
| 完了条件が検証可能 | チェックリスト形式で確認可能       |
| テストケースが記載 | 検証方法が具体的に説明されている   |
| リスクと対策が検討 | 想定リスクと対応策が記載されている |
| 親タスクの苦戦箇所が反映されているか | 親タスクのdocumentation-changelogに記録された苦戦箇所・教訓が、未タスク指示書の「実装課題と解決策」セクションまたはリスク対策に反映されている |

### 実装課題と解決策セクション（推奨）

未タスク指示書には、親タスクで遭遇した苦戦箇所と解決策を「実装課題と解決策」セクション（推奨番号: 3.5）として含めることを推奨する。

| 項目               | 説明                                                         |
| ------------------ | ------------------------------------------------------------ |
| セクション名       | `## 3.5 実装課題と解決策（親タスクからの教訓）`              |
| 記載内容           | 親タスクのdocumentation-changelogから関連する苦戦箇所を抽出  |
| 記載フォーマット   | 課題/発見経緯/解決策/教訓の4項目テーブル形式                 |
| 参照リンク         | 該当するシステム仕様書（arch-state-management.md等）へのリンク |
| 記載基準           | 未タスクの実装時に同様の問題が発生する可能性が高い場合は必須 |

**苦戦箇所を未タスクに反映するフロー**:

```
親タスクのdocumentation-changelogの苦戦箇所セクション確認
    |
    v
未タスクの実装スコープに関連する課題を抽出
    |
    v
未タスク指示書の「実装課題と解決策」セクションに転記
    |
    v
該当するシステム仕様書・06-known-pitfalls.mdへの参照リンクを追加
    |
    v
品質チェック（How/リスクと対策に苦戦箇所が反映されているか確認）
```

> **参考**: UT-STORE-HOOKS-REFACTOR-001では、合成Hookの参照不安定性（課題5.1）やESLintキャッシュ問題（課題5.2）、Phase 12更新漏れ（課題5.3）が苦戦箇所として記録された。後続の未タスク（UT-002, UT-003）にはこれらの教訓を「実装課題と解決策」セクションとして反映した。

## 指示書の品質レベル

### Level A（即実行可能）

- 100人中100人が同じ理解で実行可能
- 追加の質問なしで作業開始可能
- すべてのチェック項目を満たす

### Level B（軽微な確認必要）

- 90人以上が同じ理解で実行可能
- 1-2点の確認事項あり
- 主要なチェック項目を満たす

### Level C（要改善）

- 50人未満しか同じ理解で実行できない
- 複数の確認事項あり
- 必須チェック項目が欠落

**目標**: すべての指示書をLevel Aで作成する

## ファイル命名規則

### パターン

| タスク種別       | ファイル名パターン                  |
| ---------------- | ----------------------------------- |
| 要件系           | `requirements-{{機能領域}}.md`      |
| 改善系           | `task-{{改善領域}}-improvements.md` |
| バグ修正         | `task-{{FEATURE_NAME}}-bugfix.md`   |
| リファクタリング | `task-{{対象}}-refactoring.md`      |
| セキュリティ     | `task-{{対象}}-security.md`         |
| パフォーマンス   | `task-{{対象}}-performance.md`      |

### 命名例

```
# 良い例
requirements-auth-improvements.md
task-ui-accessibility-improvements.md
task-api-layer-refactoring.md
task-session-management-security.md

# 悪い例（曖昧）
task-fix.md
requirements.md
task-improvement.md
```

## 優先度の判定基準

| 優先度 | 基準                                                   |
| ------ | ------------------------------------------------------ |
| 高     | セキュリティ問題、ユーザー体験に大きな影響、ブロッカー |
| 中     | 品質改善、中程度のUX改善、技術的負債の解消             |
| 低     | 細かい改善、将来の拡張機能、nice-to-have               |

## 見積もり規模の判定基準

| 規模   | 目安                                      |
| ------ | ----------------------------------------- |
| 大規模 | 複数コンポーネント、3Phase以上、1週間以上 |
| 中規模 | 単一コンポーネント、2-3Phase、2-5日       |
| 小規模 | 限定的変更、1-2Phase、1日以内             |
