# 実行モード選択ガイド

> **読み込み条件**: モード選択の判断に迷った時
> **相対パス**: `references/execution-mode-guide.md`

## 概要

タスクの性質に応じて最適な実行モードを選択するためのガイドライン。

---

## モード比較表

| 観点 | Claude Code | Codex (GPT-5.2) | Gemini CLI | Claude → Codex |
|------|-------------|-----------------|------------|----------------|
| **ファイル編集** | ✅ 最適 | ❌ 不可 | ⚠️ -yフラグ時 | ⚠️ 結果のみ |
| **Git操作** | ✅ 最適 | ❌ 不可 | ❌ 不可 | ⚠️ Claude側で |
| **コードベース理解** | ✅ 深い | ⚠️ 限定的 | ⚠️ 限定的 | ✅ 深い |
| **独立分析** | ✅ 可能 | ✅ 最適 | ✅ 最適 | ✅ 可能 |
| **別視点** | ❌ 同一 | ✅ 異なる | ✅ 異なる | ✅ 異なる |
| **長文生成** | ⚠️ 制限あり | ✅ 得意 | ✅ 得意 | ✅ 得意 |
| **対話型** | ✅ 最適 | ❌ バッチ | ❌ バッチ | ⚠️ 結果確認 |
| **マルチモーダル** | ❌ テキストのみ | ❌ テキストのみ | ✅ 画像対応 | ❌ テキストのみ |
| **コスト** | サブスク内 | API課金 | 無料枠あり | API課金 |

## モード別詳細

### Claude Code（claude）

#### 最適な用途

| 用途 | 例 |
|------|-----|
| コード編集 | バグ修正、機能追加、リファクタリング |
| Git操作 | コミット、ブランチ作成、マージ |
| 対話型開発 | 段階的な実装、フィードバックループ |
| コードベース分析 | 依存関係調査、影響範囲特定 |
| テスト実行 | ユニットテスト、統合テスト |

#### 強み

- ファイルシステムへの直接アクセス
- Git CLIの完全な操作
- 対話的なフィードバック
- プロジェクト構造の深い理解

#### 弱み

- 単一の視点（Claude）
- 長文生成時のコンテキスト制限

---

### Codex（codex）

#### 最適な用途

| 用途 | 例 |
|------|-----|
| 独立分析 | コードレビュー、アーキテクチャ評価 |
| ドキュメント生成 | README、API仕様書、コメント |
| コード生成 | ボイラープレート、テンプレート |
| 別視点分析 | セカンドオピニオン、代替案提示 |
| 長文生成 | 詳細な説明文、チュートリアル |

#### 強み

- GPT-5.2の処理能力
- Claude と異なる視点
- 長文生成の得意
- バッチ処理に最適

#### 弱み

- ファイル編集不可
- Git操作不可
- コードベースの限定的な理解（コンテキスト共有時を除く）

---

### Gemini CLI（gemini）

#### 最適な用途

| 用途 | 例 |
|------|-----|
| マルチモーダル分析 | 画像付きUIレビュー、スクリーンショット分析 |
| セカンドオピニオン | Claudeとは異なるモデルでの設計検証 |
| 無料枠分析 | コスト重視の独立分析・レビュー |
| 大規模コンテキスト | 長大なドキュメントの要約・分析 |

#### 呼び出し方法

```bash
# 基本（テキスト出力）
gemini "プロンプト" -o text

# JSON出力
gemini "プロンプト" -o json

# stdinパイプ
cat context.md | gemini -o text

# 自動承認モード（ファイル操作含む）
gemini "プロンプト" -y -o text
```

#### 強み

- Google AI Studio との統合
- マルチモーダル（画像入力対応）
- 大きなコンテキストウィンドウ
- 無料枠あり

#### 弱み

- ファイル操作は `-y` フラグ必要
- Git操作不可
- 日本語精度がモデルにより異なる

---

### Claude → Codex 連携（claude-to-codex）

#### 最適な用途

| 用途 | 例 |
|------|-----|
| コンテキスト付き分析 | 既存コードを踏まえた設計提案 |
| 複合タスク | 調査 → 分析 → 提案 |
| 深い理解 + 別視点 | アーキテクチャ見直し |
| 大規模生成 | コンテキストを踏まえた大量ドキュメント |

#### ワークフロー

```
1. Claude Code: コードベース調査・コンテキスト収集
       ↓
2. Claude Code: context.md 生成
       ↓
3. Codex: コンテキストを踏まえたタスク実行
       ↓
4. Claude Code: 結果検証・統合
```

#### 強み

- 両者の強みを活用
- 深いコンテキスト + 別視点
- 大規模タスクに対応

#### 弱み

- 実行時間が長い
- コンテキスト準備のオーバーヘッド

## 判断に迷った場合

### デフォルト選択

**迷ったら `claude` を選択**

理由:
1. 対話的に方向修正できる
2. 必要に応じてCodexに切り替え可能
3. ファイル編集が必要になった場合に対応可能

### ユーザーに確認すべき観点

```
1. 「最終的にファイルを編集する必要がありますか？」
2. 「別の視点での分析が欲しいですか？」
3. 「大量のテキスト生成が必要ですか？」
```

## サブエージェント（Task agent）実行時の注意事項

### テスト実行タイムアウト（exit code 144）

サブエージェントには実行時間の制限があり、長時間のテスト実行はプロセスが SIGUSR2 で強制終了される（exit code 144）。

#### 対策

| 対策 | コマンド例 | 効果 |
|------|-----------|------|
| テスト対象を限定 | `pnpm vitest run src/renderer/components/AuthGuard/` | 実行時間を数秒に短縮 |
| 並列実行を制限 | `--no-file-parallelism` | ワーカー間の競合を回避 |
| タイムアウト明示設定 | `--testTimeout=10000` | テスト単位のタイムアウトを制御 |
| JSON出力で部分結果を保存 | `--reporter=json --outputFile=test-results.json` | 中断しても部分結果を回収可能 |

#### exit code 144 発生時のフォールバック

1. サブエージェントの出力から実行済みテスト結果を確認
2. メインエージェントで同一コマンドを再実行
3. exit code 144 を「テスト失敗」と誤認しない -- これはシグナルによる強制終了であり、テストコードの問題ではない

#### ファイル競合回避

複数のサブエージェントが同時にテストを実行する場合、以下の競合が発生しうる:

- **同一ファイルの読み書き**: 各サブエージェントのテスト対象ディレクトリを分離する
- **ポート競合**: `--poolOptions.threads.maxThreads=1` でスレッド数を制限する
- **一時ファイル競合**: `--tmpdir` オプションで一時ディレクトリを分離する

**参照パターン**: `patterns.md` 「サブエージェントでのテスト実行タイムアウト対策（exit code 144）」
